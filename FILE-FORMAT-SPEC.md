# Kyro — File Format Specification

> Version: 1.0.0
> Created: 2026-03-01
> Purpose: Canonical format for all workspace files. This document is authoritative.

Este documento define el formato exacto de cada tipo de archivo en el workspace de Kyro. Todos los parsers, serializers, y agentes de IA deben seguir esta especificación.

---

## Directory Structure

```
{KYRO_WORKSPACE_PATH}/
├── .kyro/
│   ├── config.json          # Workspace metadata
│   ├── members.json         # Global team members roster
│   └── activities.json      # Agent activity log (append-only)
└── projects/
    └── {project-slug}/
        ├── README.md        # Agent entry point (MANDATORY READ)
        ├── ROADMAP.md       # Sprint plan (generated by sprint-forge)
        ├── RE-ENTRY-PROMPTS.md  # Re-entry guide for humans/agents
        ├── sprints/
        │   └── SPRINT-NN.md # One file per sprint
        └── documents/
            └── {doc-slug}.md  # Technical documents
```

**Slugs**: lowercase, hyphen-separated, no spaces. Example: `integrar-pagos`, `sprint-01`, `arquitectura-tecnica`.

---

## 1. Workspace Config — `.kyro/config.json`

```json
{
  "id": "wks-unique-id",
  "name": "Nombre del Workspace",
  "description": "Descripción opcional del workspace",
  "createdAt": "2026-03-01T00:00:00Z",
  "updatedAt": "2026-03-01T00:00:00Z"
}
```

**Fields**:
| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | string | Yes | Unique workspace ID, format: `wks-{timestamp}` |
| `name` | string | Yes | Human-readable workspace name |
| `description` | string | No | Optional description |
| `createdAt` | ISO 8601 | Yes | Creation timestamp |
| `updatedAt` | ISO 8601 | Yes | Last modification timestamp |

## 3. Agent Activities — `.kyro/activities.json`

Append-only log. Each entry is an agent action.

```json
[
  {
    "id": "act-001",
    "projectId": "kyro",
    "actionType": "task_completed",
    "description": "Completed T1.1: Define workspace directory structure",
    "timestamp": "2026-03-01T14:00:00Z",
    "metadata": {
      "sprintId": "sprint-01",
      "taskId": "T1.1"
    }
  }
]
```

**actionType values**: `task_started` | `task_completed` | `task_blocked` | `sprint_started` | `sprint_closed` | `document_created` | `document_updated` | `project_created`

## 5. Sprint File — `projects/{slug}/sprints/SPRINT-NN.md`

El formato más importante. Las tareas usan el sistema de símbolos de checkbox.

### Task Symbol System

| Symbol | Status | Meaning |
|--------|--------|---------|
| `[ ]` | `todo` | Pendiente — no comenzado |
| `[~]` | `in_progress` | En progreso actualmente |
| `[x]` | `done` | Completado y verificado |
| `[!]` | `blocked` | Bloqueado — razón documentada inline |
| `[-]` | `skipped` | Omitido intencionalmente con justificación |
| `[>]` | `todo` (carry-over) | No completado — pasa al siguiente sprint |

> Nota: `[>]` items en el sprint actual se crean como `[ ]` en el siguiente sprint con nota "(carry-over from Sprint N)".

### Complete Sprint File Format

```markdown
---
id: sprint-01
name: Foundation Sprint
status: planned
startDate: 2026-03-01
endDate: 2026-03-08
version: 1.0.0
objective: |
  Descripción del objetivo del sprint en una o varias oraciones.
---

# Sprint 01 — Foundation Sprint

## Objetivo
Descripción amplia del objetivo del sprint y qué se espera lograr.

## Tareas

### Fase 1: Nombre de la Fase
- [x] Tarea completada con detalle
- [~] Tarea en progreso actualmente
  - Nota de progreso: avanzado hasta X
- [ ] Tarea pendiente
- [!] Tarea bloqueada — Razón: necesita que Z esté listo primero
- [-] Tarea omitida — Razón: ya no aplica porque Y
- [>] Tarea carry-over — pasa al Sprint 02

### Fase 2: Segunda Fase
- [ ] Otra tarea
- [x] Tarea completada

## Retrospectiva

### Qué funcionó bien
- Item 1
- Item 2

### Qué no funcionó
- Item 1

### Deuda técnica nueva
| ID | Item | Target |
|----|------|--------|
| D04 | Descripción del nuevo debt | Sprint N |

### Recomendaciones para el próximo sprint
1. Recomendación específica y accionable
2. Otra recomendación
```

**Frontmatter fields**:
| Field | Type | Required | Values |
|-------|------|----------|--------|
| `id` | string | Yes | Format: `sprint-NN` |
| `name` | string | Yes | Human-readable sprint name |
| `status` | string | Yes | `planned` \| `active` \| `closed` |
| `startDate` | date | No | YYYY-MM-DD |
| `endDate` | date | No | YYYY-MM-DD |
| `version` | string | No | Semver: `1.0.0` |
| `objective` | string | No | Multi-line string |

## 7. Naming Conventions

| Entity | File/Dir Name | Example |
|--------|--------------|---------|
| Workspace | Directory | `mi-workspace/` |
| Project | Directory slug | `projects/integrar-pagos/` |
| Sprint | `SPRINT-NN.md` (zero-padded) | `SPRINT-01.md`, `SPRINT-12.md` |
| Document | `{slug}.md` | `arquitectura-tecnica.md` |
| Member ID | `mbr-{n}` | `mbr-001` |
| Activity ID | `act-{timestamp}` | `act-1740873600000` |

---

## 8. IDs and Slugs

- **Slugs**: lowercase, hyphens only, no spaces, no special chars. Derived from the entity name.
  - `"Integrar Pagos"` → `integrar-pagos`
  - `"Sprint 01"` → `sprint-01`
- **IDs**: for JSON files, use prefixed format. For markdown files, the id matches the slug.
- **Project ID**: matches the directory slug (no separate ID field needed)

---

## 9. Agent Protocol

When an agent starts work on a project:

1. **Read** `projects/{slug}/README.md` — mandatory
2. **Read** `projects/{slug}/ROADMAP.md` — understand the plan
3. **Read** `projects/{slug}/sprints/SPRINT-NN.md` (latest) — current state
4. **Read** `projects/{slug}/RE-ENTRY-PROMPTS.md` — get the right prompt
5. **Write** progress: update task symbols as work proceeds (`[ ]` → `[~]` → `[x]`)
6. **Log** activity to `.kyro/activities.json`

## 10. Parsing Rules

### YAML Frontmatter
- Use `gray-matter` to parse. The library extracts frontmatter and body separately.
- All date fields are ISO 8601 strings. Parse with `new Date(str)` when needed.
- Multi-line YAML strings (using `|`) are preserved as-is.

### Task Symbol Extraction
- Tasks are lines matching the regex: `/^- \[(.)\] (.+)$/m`
- Symbol map: `' '→todo`, `'~'→in_progress`, `'x'→done`, `'!'→blocked`, `'-'→skipped`, `'>'→todo (carry-over flag)`
- Tasks may have continuation lines (indented under the task). These are notes/metadata.
- Phase grouping: tasks belong to the nearest preceding `### Phase N:` heading.

### Round-trip Guarantee
Every entity must survive a parse → serialize → parse cycle without data loss.
This is the primary correctness criterion for parsers and serializers.
