## Execution Rules

Estas reglas gobiernan todo el ciclo de vida de los sprints. Son no-negociables.

1. **Sprint-by-Sprint** — Los sprints se generan DE UNO EN UNO. Nunca pre-generar todos los sprints.
2. **Retro Feeds Forward** — La retrospectiva y recomendaciones del Sprint N-1 son entradas formales para el Sprint N.
3. **Recommendations are Dispositioned** — Cada recomendación del sprint anterior debe convertirse en tarea o tener su postergación justificada en la Disposition table.
4. **Debt is Inherited** — La tabla de Deuda Técnica Acumulada pasa de sprint en sprint íntegra. Los items nunca se eliminan.
5. **Phases are Suggested** — Las fases del roadmap son sugeridas. Durante la ejecución, fases emergentes DEBEN añadirse.
6. **Emergent Findings Generate Phases** — Si la ejecución revela issues no cubiertos por el plan, se añaden nuevas fases al sprint actual.
7. **Re-entry Prompts are Updated** — Después de INIT y de cada sprint ejecutado, los re-entry prompts se actualizan.
8. **Roadmap Adapts** — Si la ejecución revela que los sprints planeados ya no tienen sentido, este roadmap se actualiza.
9. **Definition of Done** — Cada sprint tiene criterios de completitud explícitos. Un sprint no está done hasta que todos los DoD están chequeados.

## Detailed Sprint Definitions

### Sprint 1 — File Format + Workspace Model

- **Source**: `findings/02-file-format-not-defined.md`, `findings/03-workspace-concept-missing.md`
- **Version Target**: 1.0.0
- **Type**: feature
- **Focus**: Definir el esquema canónico de archivos y añadir el tipo Workspace al modelo
- **Dependencies**: None
- **Suggested Phases**:
  1. **Spec** — Escribir `FILE-FORMAT-SPEC.md` documentando el formato canónico para cada entidad (workspace, project, sprint con task symbols, document con frontmatter)
  2. **Dependencies** — Instalar `gray-matter` (frontmatter YAML parser) y `js-yaml`; añadir `KYRO_WORKSPACE_PATH` a `.env.local`
  3. **Types** — Añadir `Workspace` interface a `lib/types.ts`; añadir config de workspace a `lib/config.ts`
  4. **Parsers** — Crear `lib/file-format/parsers.ts`: funciones para parsear sprint files, document files, project directories
  5. **Serializers** — Crear `lib/file-format/serializers.ts`: funciones para escribir TypeScript types → markdown con frontmatter
  6. **Tests** — Tests unitarios para parsers y serializers con archivos de ejemplo
  7. **Seed Files** — Crear workspace de ejemplo en disco con datos de los mocks actuales (para smoke testing)

- **Definition of Done**:
  - `FILE-FORMAT-SPEC.md` documentado y revisado
  - `lib/types.ts` tiene `Workspace` type
  - `lib/file-format/parsers.ts` parsea sprints, docs, projects
  - `lib/file-format/serializers.ts` escribe archivos válidos
  - Workspace de ejemplo existe en disco y es parseable
  - TypeScript sin errores (`npx tsc --noEmit`)

### Sprint 2 — API Routes (File I/O Bridge)

- **Source**: `findings/04-no-api-routes-for-file-io.md`
- **Version Target**: 1.1.0
- **Type**: feature
- **Focus**: Crear el puente entre el browser y el filesystem via Next.js API routes
- **Dependencies**: Sprint 1
- **Suggested Phases**:
  1. **Workspace API** — `app/api/workspace/route.ts`: GET workspace metadata, POST init workspace
  2. **Projects API** — `app/api/projects/route.ts` (list/create), `app/api/projects/[id]/route.ts` (get/update/delete)
  3. **Sprints API** — `app/api/projects/[id]/sprints/route.ts`, `/[sprintId]/route.ts`, tareas dentro del sprint
  4. **Tasks API** — CRUD de tareas dentro del sprint file, drag-and-drop status update
  5. **Documents API** — `app/api/projects/[id]/documents/route.ts`, individual doc CRUD
  6. **Security** — Validación de paths (prevenir path traversal), sanitización de IDs, error handling estándar
  7. **Integration Test** — Smoke test de todos los endpoints con `curl` o Vitest

- **Definition of Done**:
  - Todos los endpoints responden correctamente con el workspace de ejemplo (Sprint 1)
  - Validación de path traversal en todos los endpoints
  - Error handling: 404 para not found, 400 para invalid YAML, 500 para fs errors
  - TypeScript sin errores

### Sprint 4 — Store Async Adaptation + Version History

- **Source**: `findings/08-store-async-adaptation.md`, `findings/06-version-history-ephemeral.md`
- **Version Target**: 1.3.0
- **Type**: refactor
- **Focus**: Eliminar el seed síncrono, hacer todas las store actions async, git-based version history
- **Dependencies**: Sprint 3
- **Suggested Phases**:
  1. **Remove Seed** — Eliminar `getInitialState()` datos mock; el state inicial es vacío + `isInitializing: true`
  2. **Async Actions** — Convertir todas las store actions CRUD a `async` con await al service
  3. **Optimistic Updates** — Para edición de documentos y tareas: update optimista + rollback en error
  4. **Loading States** — `isSaving` state para operaciones de escritura; mejorar loading UI en `ContentRouter`
  5. **Git Init** — API route para `git init` del workspace; auto-commit en document save
  6. **Version History API** — API routes para git log y git checkout de un archivo
  7. **Version History UI** — Adaptar `VersionHistory` component para consumir git history
  8. **E2E Test** — Flujo completo: crear workspace → crear proyecto → crear sprint → añadir tareas → editar documento → ver historial

- **Definition of Done**:
  - La app arranca sin datos pre-sembrados y carga del filesystem correctamente
  - Crear/editar/borrar persiste en archivos inmediatamente
  - Document version history funciona via git
  - TypeScript sin errores
  - El flujo completo de trabajo (workspace → proyecto → sprint → tarea) funciona sin bugs
