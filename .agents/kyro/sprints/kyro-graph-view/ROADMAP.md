## Execution Rules

These rules govern the entire sprint lifecycle. They are non-negotiable.

1. **Sprint-by-Sprint** -- Sprints are generated ONE AT A TIME. Never pre-generate all sprints. Each sprint is informed by the results of the previous one.
2. **Retro Feeds Forward** -- The retrospective and recommendations of Sprint N-1 are formal inputs for Sprint N. They cannot be ignored.
3. **Recommendations are Dispositioned** -- Every recommendation from the previous sprint must either become a task or have its deferral justified in the Disposition table.
4. **Debt is Inherited** -- The Accumulated Technical Debt table passes from sprint to sprint in full. Items are never deleted, only resolved.
5. **Phases are Suggested** -- The roadmap defines suggested phases per sprint. During execution, emergent phases MUST be added when new work is discovered.
6. **Emergent Findings Generate Phases** -- If execution reveals issues not covered by the plan, new phases are added to the current sprint.
7. **Re-entry Prompts are Updated** -- After INIT and after each executed sprint, re-entry prompts are refreshed to reflect current state.
8. **Roadmap Adapts** -- If execution reveals that planned sprints need modification, this roadmap is updated. The plan serves execution, not the reverse.
9. **Definition of Done** -- Each sprint has explicit completion criteria. A sprint is not done until all DoD items are checked.

## Detailed Sprint Definitions

### Sprint 1 -- Foundation: Data Model, Parsing, Storage, Builder, API & Route

- **Source**: `findings/01-graph-data-model.md`, `findings/02-markdown-link-parsing.md`, `findings/03-sqlite-graph-storage.md`, `findings/04-graph-builder.md`, `findings/06-api-route-and-service.md`, `findings/07-route-nav-integration.md`
- **Version Target**: 4.0.0
- **Type**: feature
- **Focus**: Build the entire backend pipeline from types through API, plus the route and page skeleton. By the end of this sprint, the graph data is available via API and the page route exists (with a placeholder UI).
- **Dependencies**: None
- **Suggested Phases**:
  1. Data Model -- Define Zod schemas (GraphNode, GraphEdge, GraphCluster, GraphData) in types.ts, add UI constants to config.ts
  2. Link Parser -- Create graph-parser.ts with wiki-link, markdown-link, frontmatter-ref, and tag extraction functions
  3. SQLite Storage -- Add graph_nodes and graph_edges tables to schema, add graph query functions
  4. Graph Builder -- Create graph-builder.ts, integrate with initIndex() and reindexFile()
  5. API & Route -- Create API route, service layer integration, nav item, page skeleton

### Sprint 2 -- Interactive Graph UI

- **Source**: `findings/05-interactive-graph-ui.md`
- **Version Target**: 4.1.0
- **Type**: feature
- **Focus**: Build the interactive graph rendering using `react-force-graph-2d` (user preference), with zoom/pan controls, click-to-navigate, hover tooltips, and node dragging.
- **Dependencies**: Sprint 1
- **Library**: `react-force-graph-2d` (replaces originally planned d3-force + React SVG)
- **Suggested Phases**:
  1. Library Setup -- Install react-force-graph-2d, create graph canvas component wrapper
  2. Node & Edge Rendering -- Configure node colors (GRAPH_NODE_HEX_COLORS), edge styles, labels, dark mode
  3. Interaction Layer -- Click-to-navigate via router.push(), hover tooltips, node dragging
  4. Controls & Layout -- Zoom buttons, reset view, legend panel, responsive layout

### Sprint 3 -- Filtering, Clustering & Real-time

- **Source**: `findings/05-interactive-graph-ui.md`, `findings/06-api-route-and-service.md`
- **Version Target**: 4.2.0
- **Type**: feature
- **Focus**: Add filter panel (by type, tags), visual clustering (convex hulls or background regions), and real-time graph updates via SSE/file-watcher.
- **Dependencies**: Sprint 2
- **Suggested Phases**:
  1. Filter Panel -- Type toggles, tag filter, node search
  2. Clustering -- Visual grouping by directory, type, or tag
  3. Real-time Updates -- Integrate with useRealtimeSync for live graph refresh
  4. Minimap -- Optional overview minimap for navigation in large graphs

### Sprint 4 -- Performance, Testing & Polish

- **Source**: Retro-driven from Sprint 3
- **Version Target**: 4.3.0
- **Type**: debt
- **Focus**: Performance optimization for large projects (50+ files), comprehensive unit tests for parsers and builder, E2E tests for graph interactions, accessibility improvements.
- **Dependencies**: Sprint 3
- **Suggested Phases**:
  1. Performance -- Canvas rendering fallback, virtualization, simulation optimization
  2. Unit Tests -- Parser tests, builder tests, query tests
  3. E2E Tests -- Graph page rendering, interaction tests
  4. Accessibility -- Keyboard navigation, screen reader support, ARIA labels
